보안 항목

Warn handle_new_user() 고침

가입 시 프로필을 자동 생성하는 내부 트리거 함수가 /rest/v1/rpc/ 경로로 외부에서도 직접 호출 가능한 상태였습니다. 원래 가입 시 자동 실행만 돼야 합니다.

→ API 역할(anon·authenticated)의 실행 권한 회수. 트리거는 권한과 무관하게 동작하므로 가입 기능엔 영향 없음.

Warn prevent_profile_tier_update() 고침

요금제(free/pro)를 유저가 임의로 못 바꾸게 막는 함수의 search_path가 고정돼 있지 않아, 이론상 경로 조작 여지가 있었습니다.

→ search_path = '' 로 고정. 동작은 동일.

Warn consume / refund_analysis_credit() 설계상 정상

로그인 유저가 실행 가능한 SECURITY DEFINER 함수로 표시됐지만, 라우트가 유저 세션으로 호출하며 내부에서 auth.uid() 자기 행만 조작합니다. 권한이 필요해 유지합니다.

Info webhook_events 설계상 정상

RLS는 켜져 있지만 정책이 없다고 표시됐습니다. 이 테이블은 결제 웹훅이 service-role로만 쓰는 전용 테이블로, 일반 유저 접근을 0으로 두는 것이 의도된 상태입니다.

Warn Leaked Password Protection 직접 조치

다른 사이트에서 유출된 비밀번호(HaveIBeenPwned 대조)로 가입·변경하는 것을 막는 기능이 꺼져 있습니다. 우리는 이메일+비번 가입을 받으므로 켜는 것이 좋습니다.

→ SQL로는 불가. Auth 대시보드에서 스위치로 켜야 합니다 (아래 할 일 ① 참조).

성능 항목

Warn ×14 RLS 권한 검사 매 행 재평가 고침

"이 데이터가 네 것 맞아?"를 확인하는 규칙이 데이터 한 줄마다 auth.uid()를 다시 계산합니다. 유저·데이터가 많아질수록 느려집니다.

→ (select auth.uid()) 로 감싸 문장당 1회만 계산. 14개 정책 전부 적용, 동작 동일.

Info ×2 transactions · insights .upload_id 고침

거래·분석 데이터가 어느 업로드에 속하는지 찾을 때 색인이 없어, 업로드 삭제·조인 시 느려질 수 있습니다.

→ upload_id 커버링 인덱스 2개 추가.

Info ×2 미사용 인덱스 2건 판단 보류

한 번도 안 쓰인 인덱스로 표시됐지만, 서비스가 생성 6일차라 트래픽이 없어 "미사용"일 뿐 불필요한 건 아닙니다. 대시보드 목록 조회에 실제로 필요합니다.

→ 지금 삭제하지 않고, 실사용 데이터가 쌓인 뒤 재확인.
